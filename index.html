MG
Como consulto os dados de um colaborador pela API Suite Gen.te nuvem? Quais campos preciso enviar?
Para consultar um colaborador use a operação ConsultarPessoaPorCPFAplicandoAutorizacaoDePerfil do Serviço de Colaborador (V2):

obrigatório CpfCnpj → CPF (pessoa física) ou CNPJ (pessoa jurídica)

O response retorna um objeto PessoaV3 contendo:
• ContratosDeTrabalho — vínculos empregatícios
• Dependentes — dependentes cadastrados
• Pensionistas — pensionistas vinculados

Quer que eu gere o envelope SOAP completo?
MG
Sim, gera o XML pra mim.
Aqui está a request pronta: 
<!-- SOAP Request gerada pelo especialista em APIs -->
<soapenv:Envelope
  xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/"
  xmlns:v2="http://v2.ServicoDeColaborador/"
  xmlns:dto="http://schemas.datacontract.org/2004/07/">
  <soapenv:Header>
    <dto:LGAutenticacao>
      <dto:TokenUsuario>
        <dto:Senha>Senha123</dto:Senha>
        <dto:Usuario>usuarioteste@lg.com.br</dto:Usuario>
        <dto:GuidTenant>2288462D-A76A-42AB-82EC-4315CA68DE63</dto:GuidTenant>
      </dto:TokenUsuario>
    </dto:LGAutenticacao>
    <dto:LGContextoAmbiente>
      <dto:Ambiente>115</dto:Ambiente>
    </dto:LGContextoAmbiente>
  </soapenv:Header>
  <soapenv:Body>
    <v2:ConsultarPessoaPorCPFAplicandoAutorizacaoDePerfil>
      <FiltroDePessoaPorCPFOuCNPJ>
        <CpfCnpj>123.456.789-00</CpfCnpj>
      </FiltroDePessoaPorCPFOuCNPJ>
    </v2:ConsultarPessoaPorCPFAplicandoAutorizacaoDePerfil>
  </soapenv:Body>
</soapenv:Envelope>

O Problema

Integrar com SOAP é moroso.
E caro.

Colaboradores e clientes da suíte gen.te enfrentam barreiras técnicas e de negócio que atrasam integrações, geram erros e atrasam projetos; além de gerar demanda ao time de suporte.

🗺️

Documentação confusa

WSDLs complexos com centenas de operações, sem contexto sobre qual usar em cada situação do negócio.

❓

Campos obrigatórios ocultos

Não fica claro quais atributos são obrigatórios vs. opcionais, gerando erros de validação frustrantes.

🔗

Contexto gen.te desconhecido

Difícil entender como os campos da API se relacionam com entidades reais da suíte: ficha cadastral, estrutura organizacional, folha e etc.

⚠️

Requests mal formadas

Montar um SOAP envelope correto exige conhecimento de XML, namespaces e a forma de autenticação — gerando uma barreira de entrada alta.

🔄

Responses difíceis de parsear

Objetos aninhados e nomenclaturas técnicas tornam o mapeamento do response para dados de negócio um processo manual e lento.

💸

Alto custo de operação

Clientes muito confusos, extrapolam projetos de apoio à API e geram cada vez mais tickets ao suporte que consome tempo de especialistas — escalando o custo operacional.
